> 本文档既是项目全面的介绍说明书，也是RAG问答系统上线后的第一份测试数据。
> 上传本文件到知识库后，用户即可就以下所有内容提出问题并获得准确回答。

## 一、项目概述

### 1.1 项目名称
**团队RAG智能知识库问答助手** （Team RAG Knowledge QA）

### 1.2 项目背景
随着团队规模的扩大和项目文档的不断积累，新人入职培训成本高、知识碎片化严重、信息检索效率低已经成为团队面临的突出问题。传统的文档搜索仅能匹配关键词，无法理解用户的真正意图，且无法将多个文档的信息融合回答。

本项目旨在利用**大语言模型（LLM）** 结合 **RAG（检索增强生成）** 技术，构建一套团队级智能知识库系统，实现对项目资料、开发规范、问题排查记录的精准问答。

### 1.3 核心价值
| 价值维度 | 说明 |
|---------|------|
| 新人上手 | 新成员可直接向知识库提问，快速了解项目架构、开发规范、环境搭建流程 |
| 问题排查 | 遇到技术问题时，系统可基于历史排查记录给出准确的解决方案 |
| 知识沉淀 | 团队文档、会议纪要、开发规范统一入库，避免知识散落在个人笔记中 |
| 效率提升 | 相比手动翻阅文档，问答响应时间从分钟级降低到秒级 |

### 1.4 功能清单
1. **文档上传与入库**：支持 PDF、Word、Markdown、TXT 等格式
2. **语义分块**：基于余弦相似度的动态分块，保证每个分块语义完整
3. **混合检索**：BM25 关键词匹配 + 向量语义检索双通道，融合评分取 Top-K
4. **流式问答**：大模型逐字流式输出，体验丝滑
5. **多轮对话**：基于 Redis 的短期记忆管理，支持上下文连续追问
6. **Query Rewriting**：上下文聚合改写查询，提升检索召回率
7. **知识库管理**：查看已上传文档，按类型分类管理

## 三、技术架构说明

### 3.1 整体架构
```
用户浏览器 (React)
    │
    │ HTTP / SSE
    ▼
Spring Boot 后端 (8080)
    │
    ├── ChatMemoryService (Redis 短期记忆)
    │       │
    │       └── 滑动窗口管理 + Query Rewriting
    │
    ├── RagQAService (混合检索 + 大模型问答)
    │       │
    │       ├── BM25Retriever (关键词检索)
    │       │
    │       └── PineconeEmbeddingStore (向量语义检索)
    │               │
    │               └── text-embedding-v3 向量化
    │
    ├── DocumentService (文档上传 + 语义分块 + 入库)
    │       │
    │       ├── SemanticSplitter (语义分块)
    │       │
    │       └── EmbeddingModel → Pinecone
    │
    └── StreamingChatModel (Qwen-plus 流式输出)
```

### 3.2 技术选型一览
| 组件 | 选型 | 版本 | 用途 |
|------|------|------|------|
| 后端框架 | Spring Boot | 3.2.6 | Web 服务 + REST API |
| AI框架 | LangChain4j | 1.0.0-beta3 | LLM交互、RAG流程编排 |
| 向量模型 | text-embedding-v3 | — | 阿里百炼，1024维向量化 |
| 向量数据库 | Pinecone | — | 向量存储与语义检索 |
| 大模型 | Qwen-plus | 3.5-Plus | 通义千问，397B参数（MoE激活17B） |
| 短期记忆 | Redis | 7.x | 会话上下文管理 |
| 前端 | React + TypeScript | 18.x | 单页应用 |
| 构建工具 | Vite | 8.x | 前端快速构建 |
| API文档 | Knife4j | 4.3.0 | OpenAPI3 在线接口测试 |

### 3.3 核心流程（问答服务）
1. **用户输入问题** → 前端通过 SSE 请求 `/api/rag/qa`
2. **召回短期记忆** → 从 Redis 获取本次会话最近 10 轮对话
3. **Query Rewriting** → 将历史上下文与当前问题聚合改写
4. **混合检索**
   - BM25 关键词检索：适配专业术语、代码命令精确匹配
   - 向量语义检索：通过 Pinecone 进行语义相似度匹配
   - 融合得分，取 Top-3 条最相关内容
5. **提示词构建** → 融合知识库检索内容 + 对话历史 + 用户问题
6. **调用大模型** → Qwen-plus 流式输出答案
7. **更新短期记忆** → 将本轮问答存入 Redis

## 四、开发规范

### 4.1 Java 后端编码规范

#### 命名规范
- 类名：大驼峰命名，如 `DocumentService`、`RagQAController`
- 方法名：小驼峰命名，如 `uploadAndStore()`、`vectorRetrieve()`
- 常量：全大写下划线，如 `MEMORY_KEY_PREFIX`、`WINDOW_SIZE`
- 包名：全小写，如 `com.team.rag.config`

#### 注释规范
- 类级别必须有 Javadoc 注释，说明类用途
- 核心方法必须有 `@param` 和 `@return` 注释
- 复杂算法逻辑需要行内注释说明步骤

#### 异常处理
- Controller 层统一返回结果，不抛出运行时异常
- Service 层内部使用 try-catch 捕获异常，并记录 ERROR 日志
- 不允许 catch 到异常后空处理（swallow exception）

#### 依赖注入
- 优先使用构造器注入（`@RequiredArgsConstructor`），不使用 `@Autowired` 字段注入
- 配置值使用 `@Value` 注入，统一管理在 `application.properties`

### 4.2 前端编码规范

#### 组件规范
- 使用函数式组件 + React Hooks
- 每个组件一个文件，文件名与组件名一致（PascalCase）
- Props 必须定义 TypeScript 接口

#### 样式规范
- 使用原生 CSS，不使用 CSS-in-JS 或 Tailwind
- 所有颜色、间距、字号使用 CSS 变量（Design Token）
- 动画使用 `transition` 和 `@keyframes`，避免 JS 动画

#### API 调用
- 统一封装在 `src/api/` 目录
- 使用 `fetch` 原生 API，不引入 axios
- 流式请求使用 `ReadableStream` 消费 SSE 数据

## 五、常见技术问题排查

### 5.1 Redis 缓存穿透

**问题描述**：大量请求查询一个不存在的 Key，直接打穿 Redis 到达数据库。

**解决方案**：
1. **布隆过滤器**：在 Redis 前加布隆过滤器，判断 Key 是否可能存在
2. **缓存空值**：对不存在的 Key 缓存空值，设置较短的过期时间（如 5 分钟）
3. **参数校验**：接口层对异常参数（如负数ID、空字符串）直接拦截

**代码示例**：
```java
// 缓存空值策略
public String getFromCache(String key) {
    String value = redisTemplate.opsForValue().get(key);
    if ("NULL_PLACEHOLDER".equals(value)) {
        return null; // 缓存了空值，直接返回
    }
    if (value == null) {
        // 查询数据库
        String dbValue = queryFromDB(key);
        if (dbValue == null) {
            // 缓存空值，防止缓存穿透
            redisTemplate.opsForValue().set(key, "NULL_PLACEHOLDER", 5, TimeUnit.MINUTES);
            return null;
        }
        redisTemplate.opsForValue().set(key, dbValue, 30, TimeUnit.MINUTES);
        return dbValue;
    }
    return value;
}
```

### 5.2 Redis 缓存雪崩

**问题描述**：大量缓存同时过期，导致大量请求同时到达数据库。

**解决方案**：
1. **过期时间加随机值**：为缓存的过期时间添加随机偏移量（如 TTL + random(0, 300) 秒）
2. **多级缓存**：本地缓存（Caffeine）+ Redis 缓存，双层保护
3. **限流降级**：使用 Sentinel 限流，保护后端数据库

### 5.3 Spring Boot 启动报错 - DataSource 未配置

**问题描述**：启动时报 `Failed to configure a DataSource: 'url' attribute is not specified`

**解决方案**：
如果项目暂未使用关系型数据库，在启动类添加排除注解：
```java
@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)
public class TeamRagApplication { ... }
```

### 5.4 Pinecone 向量检索返回空结果

**问题描述**：已上传文档，但问答检索不到任何内容。

**排查步骤**：
1. 检查 Pinecone Index 维度是否为 1024（与 text-embedding-v3 一致）
2. 检查环境变量 `PINECONE_INDEX` 和 `PINECONE_REGION` 是否正确
3. 检查命名空间 `pinecone.namespace` 配置是否一致
4. 登录 Pinecone 控制台，确认 Index 中是否有向量数据
5. 查看后端日志 `com.team.rag` 级别的 DEBUG 日志，定位是否有异常

### 5.5 前端流式输出不显示

**问题描述**：发送问题后，前端不显示 AI 回复或显示不完整。

**排查步骤**：
1. 打开浏览器 F12 开发者工具 → Network 面板，查看 `/api/rag/qa` 请求
2. 确认请求的 Content-Type 是否为 `text/event-stream`
3. 确认后端 CORS 配置允许前端域名（`http://localhost:5173`）
4. 查看后端日志是否有大模型调用失败的 ERROR 日志
5. 检查 `DASH_SCOPE_API_KEY` 环境变量是否配置正确

### 5.6 文档上传失败 - 文件过大

**问题描述**：上传 PDF 时报错 `Maximum upload size exceeded`

**解决方案**：
在 `application.properties` 中调整文件上传大小限制：
```properties
spring.servlet.multipart.max-file-size=50MB
spring.servlet.multipart.max-request-size=50MB
```

### 5.7 Query Rewriting 导致检索偏移

**问题描述**：多轮对话后，检索结果与当前问题不相关。

**解决方案**：
1. 当前实现取最近 3 轮对话作为上下文，过多上下文会引入噪声
2. 可在 `ChatMemoryService.rewriteQuery()` 中减少上下文轮数
3. BM25 检索使用原始查询词（不经过改写），向量检索使用改写后的查询

## 六、环境搭建指南（新人必读）

### 6.1 开发环境要求
- **JDK**: 17 或以上版本
- **Maven**: 3.9 或以上版本
- **Node.js**: 18 或以上版本
- **Redis**: 7.x（本地运行）
- **IDE**: IntelliJ IDEA 推荐（需安装 Lombok 插件）

### 6.2 环境搭建步骤

#### 步骤一：克隆项目
```bash
git clone https://gitlab.team.com/team-rag-knowledge.git
cd team-rag-knowledge
```

#### 步骤二：配置环境变量
参考项目根目录下的 `环境变量配置指南.md`，配置以下 4 个核心环境变量：
- `DASH_SCOPE_API_KEY`
- `PINECONE_API_KEY`
- `PINECONE_INDEX`
- `PINECONE_REGION`

#### 步骤三：启动 Redis
```bash
redis-server
```

#### 步骤四：启动后端
```bash
cd backend
mvn spring-boot:run
```
启动成功后访问 `http://localhost:8080/doc.html` 查看 API 文档。

#### 步骤五：启动前端
```bash
cd frontend
npm install
npm run dev
```
启动成功后访问 `http://localhost:5173` 即可使用。

#### 步骤六：上传测试文档
1. 点击侧边栏底部的「上传知识文档」按钮
2. 选择本文件（`项目介绍与测试文档.md`）
3. 文档类型选择「项目资料」
4. 点击「上传并入库」
5. 等待分块和向量化完成

#### 步骤七：开始提问
尝试以下问题测试知识库效果：
- "我们团队有哪些成员？各自负责什么？"
- "Redis 缓存穿透如何解决？"
- "项目使用了哪些技术栈？"
- "新人如何搭建开发环境？"
- "后端 Java 编码规范有哪些要求？"
