核心技术选型
1. 大模型开发框架：Langchain4j     
  a. 备注：和 SpringAl 底层原理都是一样的，用哪个无所谓
2. 向量模型：text-embedding-v3    
  a. 备注：为什么选这个，请看《不同 Embedding 模型的对比与分析》
3. 向量数据库：Pinecone
  a. 备注：为什么选这个，请看《混合检索实现规划》
4. 文档解析：Apache Tika（支持多格式文档文本提取）
5. 大模型(LLM):  Qwen-plus
千问 3.5-Plus（Qwen3.5-Plus）的参数规模如下：
● 总参数量：397B（3970 亿参数）
● 激活参数量：17B（170 亿参数）
它采用稀疏 MoE 架构，推理时仅激活约 5% 的参数，兼顾大模型能力与高效推理。



文档流程
1. 读取指定文件夹下的md文件
2. 使用  [ 文档解析：Apache Tika（支持多格式文档文本提取）]   将文档解析 为 Document
3. 为 Document 添加元数据/标签信息（目的：提升后续的检索召回效果）
4. 基于语义分块策略，将Document 分块成 List<TextSegment>
5. 把 List<TextSegment> （利用embedding model text-embedding-v3 ） 向量化
6. 把 向量化后的内容和原始文本 存入向量数据库 Pinecone


问答服务 && 问答流程
团队智能知识库系统  问答的过程 
1. 用户在会话中输入内容/问题
-- 后端处理过程
2. 召回短期记忆（从Redis中获取本次会话的短期记忆，并且有滑动窗口+摘要的机制）
3. 查询重写:   Query Rewriting （上下文聚合 + 改写）
4. 召回长期记忆（将查询重写后的内容向量化，去向量数据库，搜索用户主动录入的长期记忆）
5. RAG 过程
  a. 将查询重写后的内容向量化，基于向量相似度 (即语义匹配) 去向量数据库 Pinecone 中检索内容
  b. 基于 BM25 (关键词匹配) 去 Pinecone 中检索内容
  c. 这两个检索过程 Pinecone 会帮我们一起做的，并按权重融合得分，然后取top k 条数据
6. 提示词构建：融合长短记忆+RAG检索内容+用户输入为提示词
7. 调用大模型(LLM)，接受大模型返回内容
8. 更新短期记忆（即本次会话的内容）
