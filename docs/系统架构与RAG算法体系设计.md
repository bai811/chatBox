# 系统架构与 RAG 算法体系设计

本文档适合 **后端开发工程师**、**算法工程师** 和 **系统架构师** 阅读，旨在全方位解析整个知识库问答机器人的逻辑串联体系。

## 1. 核心架构与业务流

### 1.1 大一统流程交互架构

系统主线交互流程采用的是带有短周期对话树的增强版 RAG 流：

1. **会话拦截与记忆管理机制**
   - 依赖项：`ChatMemoryService`、Redis。
   - 用户的 Request 首先经由 SessionId 进入缓存层。将用户的新提问插入历史记录链条尾部。若对话超长（目前限制为取最后若干轮历史记录），系统会使用固定窗口或 Token 计算量剥离早期上下文。

2. **Query 意图改写（Query Rewriting）**
   - RAG 检索最大的软肋之一在于“指代不明”。例如："那它怎么解决？"。
   - 我们的业务模块利用 `LangChain4j` 自动触发一段极其轻量的前置模型调用环节：让大模型结合短期记忆把原 Query 改写为带有明确主语的实体意图，如："系统遇到的缓存穿透问题如何解决？"。然后进入检索层。

3. **智能双路融合检索（Hybrid Retrieval）**
   - 左路检索（BM25）：偏重关键词精准命中，擅长专有名词、API 命令的直接检索。
   - 右路检索（Pinecone Vector）：将改写后的用户问题 Embedding 成 1024 维的高密向量进行 KNN 相似度检索。
   - 双路合并与排序：对左右双路的 Segment 取并集，利用综合阈值（阈值 `vectorMinScore=0.8`）去粗取精，摘录含金量最高的 Top K 返回作为上下文片段。

4. **主模型流式答卷生成**
   - 将“RAG切片集群”、“短期记忆列表”、“安全指令提示词 Prompt” 灌入主问答大模型（通过 API 调用 Qwen-plus）。主模型利用 SSE 管道 `Flux<String>` 实现逐字流式打字机吐出结果，以最大程度抹除生成时的长请求等待时间。

## 2. 向量分块算法深度解析 (Semantic Splitter)

长篇宏论如果不切块，不但会把上下文和 Token 额度撑爆，还会引起“大海捞针陷阱”。

### 2.1 基于 NLP 句法深度的 SemanticSplitter

项目采用动态的语义分隔，规避了传统的“按字数一刀切（Token Text Splitter）”造成句子被腰斩引发的语义截断问题。

```xml
# application.properties 中的精细控制
rag.semantic-splitter.similarity-threshold=0.8
rag.semantic-splitter.max-tokens=1024
rag.semantic-splitter.overlap-tokens=50
```

- **Overlap (滑动视窗缓冲)**
  - 规定 `overlap-tokens=50`，这保证了上一块文档的末尾与下一块文档的开头有语义粘接层。能完美保障哪怕答案正好切在两个 Chunk 的缝隙间，最终在合并 Prompt 阶段仍逻辑闭环。

## 3. 防幻觉 Prompt 隔离仓机制

在 `RagQAService.java` 中，有一行强制安全拦截器 Prompt 规则：
> `只能根据知识库内容进行提取和总结。如果提供的知识库内容只有此问题本身而没有具体的答案，或者完全是不相关内容，请直接回复：“抱歉，知识库中未找到相关答案。”`

这是系统的坚守底线。
它严防知识库因为受到“越狱提问攻击”而胡乱向外散布未经求证的“事实”。由于 RAG 系统经常面临不规范测试问题，所以我们不仅要求防幻觉，甚至特别禁止大模型反馈诸如“片段中未见到、步骤几只有空标题”这样的底层内部逻辑，而是强制转换为直白的业务术语，大幅优化了 C 端的体验边界。

## 4. 后续演进点（Future Work）
- **Reranker 机制的引入**：在本地小规模使用如 `BGE-Reranker` 模型对 BM25 和 Vector 取出的 20 条结果在此阶段重新进行更加精密的交叉重排，最终取 Top-3 返回给大模型从而进一步收缩上下文成本并拉高精度准确率。
- **多模态拓展**：引入可解析表格结构或者 PDF 中流状图像 OCR 甚至大模型 Vision API，处理无法变成纯文本结构的重度报表文件。
